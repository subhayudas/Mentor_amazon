## 1. Analytics Dashboard - Geographic Statistics

**Requirements:**
- Add country-based statistics displaying the geographic distribution of mentors and mentees
- Show data in two separate views/sections:
  - Mentor locations by country (with counts)
  - Mentee locations by country (with counts)
- Include visual representation (charts/maps recommended)
- Ensure data updates in real-time as new users register

**Technical Notes:**
- Store country data during user registration
- Implement data aggregation queries efficiently
- Consider using a library like Chart.js or Recharts for visualization

## 2. Mentor Signup Form - Mentorship Preference

**Requirements:**
- Add a new required field asking mentors to select their preferred mentorship model:
  - Option 1: "Ongoing mentorship with one mentee" (long-term commitment)
  - Option 2: "Different mentees each session" (flexible, rotating mentees)
- Make this a clear radio button or dropdown selection
- Store this preference in the mentor's profile
- Use this data in the matching algorithm

**Validation:**
- Field must be required (cannot submit without selection)
- Display clear descriptions for each option so mentors understand the commitment

## 6. Mentor Session Approval System (High Priority, Complex)

**Requirements:**
- Implement a session request workflow:
  1. Mentee requests a session with a mentor
  2. Mentor receives notification of the request
  3. Mentor can view session details (date, time, mentee info, topic)
  4. Mentor has options to:
     - **Accept** - Session is confirmed and added to both calendars
     - **Reject** - Mentee is notified, session not scheduled
     - **Propose alternative time** (optional enhancement)
  5. Only after mentor acceptance does the session appear on calendars

**Technical Implementation:**
- Create a "Pending Sessions" view for mentors
- Add notification system (email/in-app) when new session requested
- Implement session status tracking (pending/accepted/rejected)
- Update calendar integration only after acceptance
- Send confirmation notifications to both parties
- Handle timeout scenarios (if mentor doesn't respond within X days)

**User Experience:**
- Clear notification badges for pending requests
- Easy-to-use accept/reject interface
- Show all relevant session information before decision
- Provide feedback messages after each action

**Fallback Plan:**
- If too complex initially, document the feature clearly for Phase 2
- At minimum, implement email notifications to mentors about new bookings with session details

## CRITICAL: Implementation Constraints

**NON-NEGOTIABLE REQUIREMENTS:**

1. **DO NOT break existing functionality** - All current features must continue working exactly as they do now
2. **DO NOT change the core architecture** - Keep the existing tech stack and structure
3. **Make minimal, surgical changes** - Add new features without refactoring existing code
4. **Test existing features after each change** - Ensure nothing breaks
5. **Keep the app runnable at all times** - Never leave it in a non-functional state

## Technical Decisions - Use Existing Solutions

For any unknowns, **use whatever is already in the codebase**:

- **Country list**: Use a standard library (like country-list npm package) or simple dropdown with major countries
- **Calendar system**: If calendar integration exists, extend it. If not, start with a simple database-stored schedule
- **Notifications**: If email is set up, use that. If nothing exists, start with in-app notifications only (simplest)
- **i18n implementation**: Use the most popular library for your framework (react-i18next for React, vue-i18n for Vue, etc.)
- **Admin content editing**: For now, just hardcode the contextual text in a separate config file that can be easily edited later

## Development Approach

1. **Analyze the existing codebase first** - Understand what's already there before adding anything
2. **Add features incrementally** - One at a time, test, commit, then move to next
3. **Use feature flags if possible** - So new features can be turned off if they cause issues
4. **Keep backups/commits frequent** - So you can roll back if something breaks
5. **Default to the simplest working solution** - Don't over-engineer

## If You Hit a Blocker

- **Don't guess** - Ask for clarification if something is unclear about existing code
- **Don't break things to add features** - If a feature would require major changes, flag it and suggest a simpler alternative
- **Provide options** - If you're unsure about implementation approach, present 2-3 options with pros/cons

The goal is **new features that integrate smoothly**, not a rewrite. Preserve what works, add what's needed.
